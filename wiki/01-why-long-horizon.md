# 01 · 为什么要做"长时程"基准

## 短任务测不出来的东西

主流 LLM 评测大多是"一道题一道题"的：给一段题面，让模型给一个答案，然后打分。这类评测能很好地衡量"知识广度"和"单步推理"，但**测不出来一类很关键的能力**——一个模型能不能在一个**持续变化、决策互相影响**的世界里稳定地做对事。

举几个例子：

- **会写代码 ≠ 会维护项目**：你能解 100 道 LeetCode，不代表你能在一个长期项目里始终保持代码风格一致、不互相打架。
- **能算清单题 ≠ 会做长期预算**：你能算出"这单赚多少"，不代表你能管好"未来三个月每月够不够发工资"。
- **能识别一张图里的猫 ≠ 能从行为里识别坏人**：单帧识别和"看一长串行为里谁有问题"是两种能力。

j-yc-bench 想做的就是**把这种"持续滚动、决策互相影响、信息不完全"的环境**抽象成一个可重复运行的基准。

## "经营一年"提供的压力

为什么一定要让 Agent 跑这么久？因为只有在长时程里，下面这些压力才会真正显现出来：

### 1. 决策的"复利"

早期一个看似不重要的选择——比如"每次都把 8 个员工全堆到一个任务上"——会让所有人的工资按 tier 中位数稳定上涨（成功完成任务会触发 `salary_bump_pct = 1%` 的固定加薪），但**实际只有前 4 个人在贡献**（Brooks's Law 截断）。**几个月后**，你才会发现工资负担已经把跑道压垮，而队伍的实际产能并没有相应增长。

短任务里，这种"复利效应"根本来不及发生。

### 2. 时间约束的真实化

- 工作只在工作时间（工作日 9:00–18:00，每天 9 小时）发生。周末、夜晚不算。
- 工资每月**第一个工作日**自动扣一次，不管你赚没赚。
- 截止日期是从你"接单"那一刻开始算的，**规划时间、派人时间也会消耗截止日期**。
- 在 `default` 预设下，信任有缓慢的被动衰减（`trust_decay_per_day = 0.009`），声望默认无被动衰减——但只要你延期或取消任务就会扣声望，长期不接活的成本仍然是真实的（账上钱被工资啃光）。

在一道单步题里，这些都没意义。在一年的尺度上，它们会把"看起来都行"的策略筛掉一大半。

### 3. 上下文窗口逼出"外部记忆"

LLM 上下文是有限的。Agent 循环只保留**最近 20 轮**对话（`history_keep_rounds = 20`），更早的会被截断。

这意味着——

> 如果 Agent 不主动把"我在第 5 回合接 `Cipher Corp` 的两个任务都延期了，疑似 RAT" 这种观察写到 `scratchpad` 持久备忘录里，等到第 50 回合，它已经完全忘了，可能又会去接 `Cipher Corp` 的活。

scratchpad 内容会被注入到系统 prompt 里、**不受 20 轮截断影响**。**长时程基准会强制 Agent 学会用工具维护自己的长期记忆**，而不是依赖上下文。这是单步任务里测不到的。

### 4. 信息不完全 + 对抗性环境

- 员工的每个领域速率对 Agent 是**公开**的（`employee list` 直接返回 `skill_rates`）——这一项不构成信息劣势。
- 客户的"忠诚度"（是不是 RAT 客户）对 Agent 是**隐藏**的。
- 接受任务后的 scope creep（工作量被偷偷膨胀）也是隐藏的——但 Agent 可以通过对比"`market browse` 时记下的 required_qty"与"`task inspect` 看到的当前 required_qty"来识别。

Agent 必须通过"接活 → 看结果（按时 / 延期）→ 推断 → 调整"的反复循环来识别 6 个客户里哪几个是 RAT（默认 35%，至少 1 个）。这个推断过程**至少要十几个客户级观测样本才能稳定**。在短任务里你根本攒不出样本。

## 它为什么不是"游戏化"，而是基准

很多人看到"模拟创业"会以为这是个有趣的演示玩具。但 j-yc-bench 之所以是基准，关键在三件事：

1. **确定性**：同一个 seed → 同一个世界 → 同一套事件顺序。可以严格对比。
2. **机制可审计**：每一笔钱、每一次工作量推进、每一个事件，都在 SQLite 里可查。不存在"模型说自己赚了 1 万实际只赚了 100"的情况。
3. **统一接口**：所有模型通过完全一样的 CLI 接入，**没有模型特有的提示词技巧**作为变量。

也就是说，模型 A 和模型 B 在 j-yc-bench 上的分数差异，**真的来自能力差异**，而不是"谁的 prompt 写得花哨"。

## 它能告诉你什么、不能告诉你什么

**它适合回答这些问题：**

- 模型 X 在持续决策任务上是不是比模型 Y 更稳？
- 模型在多大程度上能识别隐藏模式（如 RAT 客户）？
- 模型会不会主动维护 scratchpad 长期笔记？
- 模型在压力下（低跑道、临近月初工资）会怎么权衡？
- 不同温度、不同 prompt 风格对长期表现的影响？

**它不太适合回答这些问题：**

- 模型的纯粹知识广度（这是 MMLU、GPQA 之类该做的事）。
- 模型的代码细节能力（这是 SWE-bench 之类该做的事）。
- 模型对单步复杂推理的能力（这是 ARC、AIME 之类该做的事）。

j-yc-bench 不取代它们；它补上一块**"长期执行一致性"**的空白。

## 跟"开放世界 Agent 评测"的关系

| 开放世界基准 | j-yc-bench |
|--------------|------------|
| 动作空间几乎无限 | 动作空间是一组 CLI 命令，明确闭合 |
| 难以判定"做对没有" | 有非常清晰的指标（资金、声望、是否破产） |
| 多次运行结果难对比 | 给定 seed 完全可复现 |
| 容易被"刷数据" | 隐藏信息 + 对抗性客户 + 复利让"刷"变得不容易 |

简单说：**j-yc-bench 牺牲了一些"自由度"，换来了"可比较"和"可复现"**。这对一个基准来说是值得的取舍。

## 一句话总结

> 想看一个模型"短时间能不能想到聪明答案"，用其他基准。
> 想看一个模型"长时间能不能把聪明答案兑现成可持续的结果"，用 j-yc-bench。
