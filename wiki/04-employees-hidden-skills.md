# 04 · 员工与隐藏技能

## 你的 8 人小团队

游戏开局（以 `default` 预设为例），公司有 **8 名员工**（由 `num_employees` 决定）。

代码里 tier 用一个 10 人模板生成：`[J, J, J, J, J, M, M, M, S, S]`（5 junior、3 mid、2 senior）。当 `num_employees < 10` 时，会按模板取前 N 个再随机洗牌，所以**默认 8 人的实际 tier 比例不是固定的**——某一局可能 4J/3M/1S，另一局可能 5J/2M/1S。

CLI 里**没有招聘 / 解雇命令**——你这一年就是带这 8 个人，谁都开不掉、也加不进新人。

> 因为员工和客户用一个**固定的 world seed = 1** 生成，所以**所有 run seed 下的同一支队伍是一样的**。变化的只有任务市场。

## 你能从 `employee list` 看到什么

`java -jar j-yc-bench.jar employee list` 返回的是：

```json
{
  "count": 8,
  "employees": [
    {
      "name": "Emp_3",
      "tier": "senior",
      "salary_cents": 1250000,
      "skill_rates": {
        "research": 8.42,
        "inference": 7.91,
        "training": 9.05,
        "data_environment": 7.33
      },
      "active_tasks": ["Task-17", "Task-29"]
    }
  ]
}
```

**在当前实现里 Agent 能直接看到每个员工每个领域的速率**。这并不意味着"分配最优员工"就是 trivial 的——它仍然要和**吞吐量切分、Brooks's Law、工资上涨速度、跨任务并行**这些机制一起权衡。

但下面这两件事，Agent **仍然不能**直接看到：
- 客户的隐藏忠诚度（是否 RAT）。
- 一个任务在你接受**之后**会不会被 scope creep 偷偷把工作量膨胀掉。

## 速率分布（`default` 预设）

每个员工的每个领域速率，是**独立**从该 tier 对应的均匀区间采样出来的：

| Tier | 速率区间（单位 / 小时） | 月薪区间 |
|------|--------------------------|---------|
| junior | `uniform(1.0, 4.0)` | $2,000 – $4,000 |
| mid | `uniform(4.0, 7.0)` | $6,000 – $8,000 |
| senior | `uniform(7.0, 10.0)` | $10,000 – $15,000 |

**关键点**：每个领域**独立采样**。一个 senior 完全可能在 training 上接近 10，但在 research 上只有 7。

## 工资

所有员工的工资**每月第一个工作日**统一从公司账上扣一次。**和你接没接活、做不做事无关，工资照发**。每位员工有 `salary_max_cents = $100,000/mo` 的硬上限。

## 工资会涨（很重要）

每**成功**完成一个任务，**所有被指派到该任务的员工**都会按 tier 中位数固定额度加薪：

```
工资增量 = tier 中位数月薪 × salary_bump_pct
```

`default` 预设里 `salary_bump_pct = 0.01`（1%）。具体增量：

- junior（中位数 $3,000）：每次成功 +$30/月
- mid（中位数 $7,000）：每次 +$70/月
- senior（中位数 $12,500）：每次 +$125/月

**这是按 tier 中位数算的固定值，不是复利**。但这个增量是**永久叠加**到月薪上的。注意：**只有按时完成才涨薪**，延期完成不涨。

## 技能成长

任务成功完成时，**部分**被指派员工会获得速率加成：

```
新速率 = 旧速率 × (1 + skill_boost_pct)
```

`skill_boost_pct` 从 `normal(mean=0.12, std=0.06)` 截断到 [0.01, 0.40] 采样——平均 +12%。速率硬上限 `skill_rate_max = 10.0`。

**但不是所有人都能拿到加成**。对该任务的每个领域，按速率排序，**只有前 4 名（`EFFICIENT_TEAM_SIZE = 4`）拿到加成**，第 5 人开始不奖励技能。

## 吞吐量切分（最容易被忽视的规则）

如果一个员工**同时在 N 个活跃任务上**，那么他在**每个任务**上的有效速率会被切成 `base / N`：

```
有效速率 = 基础速率 / 该员工的活跃任务数
```

举例：Emp_5 的 research 基础速率是 6.0/小时。
- 只做 1 个任务：贡献 6.0/小时
- 同时做 3 个任务：每个上面只贡献 2.0/小时

**总产能保持不变**——切分既不奖励也不惩罚并行，只是"摊薄"。

## "多人一任务"怎么算（含 Brooks's Law）

一个任务可以同时指派多个员工。对**每个领域**单独计算：

1. 拿出所有指派员工在该领域的"切分后速率"（`base / 活跃任务数`）。
2. **从大到小排序**。
3. **前 4 个人**（`EFFICIENT_TEAM_SIZE = 4`）按完整速率累加。
4. **第 5 个及以后**乘以 `OVERCROWD_PENALTY = 0`，也就是**完全不贡献**。

任务完成时间按所有领域的瓶颈：

```
任务完成时间 = max( 该领域剩余工作量 / 该领域有效速率 )  # 跨所有领域取最大值
```

**最慢的领域决定完成时间**。如果一个任务需要 60 单位 research + 30 单位 training，你派了 4 个 research 强的、0 个 training 强的，**任务会卡在 training**。

### 核心张力表

| 策略 | 好处 | 代价 |
|------|------|------|
| 聚焦：1 员工 1 任务 | 那个员工 100% 算他的速率 | 同一时间只在推进 1 个任务 |
| 并行：1 员工 N 任务 | 同时推进 N 个 | 每个上面只算 base/N |
| 大团队：N > 4 人上同一个任务 | （没好处） | 第 5 人之后**零贡献**，还浪费吞吐量 |

## 实战指南

### 早期（前 1~2 个月）

- 直接读 `employee list` 的 `skill_rates`，按领域强项对应任务派人。
- 一个任务最多派 4 人（Brooks's Law 截断）。
- 关注 `active_tasks` 字段——它告诉你这个员工吞吐量被切成几分之几。

### 中期

- 优先并行 2~4 个任务，每个用 1~3 人小队覆盖。
- 想培养弱者？把他塞进"高手不太多"的任务里才学得到（只有前 4 名拿技能加成）。

### 后期

- 已"练满"（速率达到 10.0）的员工，每次再派他做该领域任务时**技能零边际收益**，但工资仍会永久叠加。
- 此时优先让有成长空间的员工做，避免月度工资被无效叠加吃光跑道。
- 工资到 `$100K/mo` 上限后才会停止增长。

## 一个反直觉的小结论

> "把所有人都堆到最重要的任务上"听起来天经地义，但在 j-yc-bench 里，**5 个及以上的团队是反模式**：
>
> - 第 5 人开始对该任务零贡献（`OVERCROWD_PENALTY = 0`）。
> - 这第 5 人在自己其它任务上的吞吐量却被切了。
> - 任务成功后他还和前 4 个人一起涨工资。
>
> 也就是说，"全员爆肝一单"对该任务无加速、对全局有拖累、对成本有放大——三重亏。聪明的策略是"用 1~4 人小队覆盖多条并行任务线"。
